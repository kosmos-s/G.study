# 문제 해결 기록

이 문서는 GPT Roguelike 개발 중 실제로 발생한 오류와 확인 과정을 기록합니다.

확인된 원인, 수정 시도, 해결 여부를 구분하며 재검증이 끝나지 않은 항목은 해결 완료로 표시하지 않습니다.

## 몬스터가 로비에 생성되는 문제

### 문제

생존 전투가 시작되기 전이거나 잘못된 위치를 참조할 때 몬스터가 전투 맵이 아닌 로비 주변에 생성되었습니다.

### 확인 과정

- 몬스터 생성 코드가 참조하는 SpawnPoint 확인
- 로비와 생존 맵의 생성 위치 분리 여부 확인
- 생존 세션 시작 상태와 스폰 조건 확인
- 몬스터가 추적하는 플레이어와 이동 범위 확인

### 대응

전투 지역 전용 생성 위치를 사용하고, 생존 세션이 시작된 상태에서만 몬스터를 생성하도록 구조를 정리했습니다.

### 상태

수정 적용 후 기본 테스트에서는 로비와 전투 지역이 분리되었으나, 전체 회귀 테스트에서 재발 여부를 계속 확인합니다.

## DataStore StudioAccessToApisNotAllowed

### 문제

```text
DataStoreService: StudioAccessToApisNotAllowed
Cannot write to DataStore from studio
```

Studio 테스트에서 플레이어 데이터를 저장할 수 없었습니다.

### 원인

Roblox Studio의 API 서비스 접근 설정이 비활성화된 상태였습니다. 저장 로직 자체의 오류와 개발 환경 설정 문제를 구분해야 했습니다.

### 해결

```text
Game Settings
→ Security
→ Enable Studio Access to API Services
```

설정을 점검하고 Studio 테스트 환경에서 DataStore 접근을 허용했습니다.

### 상태

설정 오류의 원인은 확인했습니다. 재접속 후 직업과 장착 무기가 정확히 복원되는지는 최신 패치에서 별도로 재검증합니다.

## WaitForChild Infinite Yield

### 문제

다음 객체를 기다리는 과정에서 Infinite Yield 경고가 발생했습니다.

- `PickupManager`
- `RemoteEvents`
- `ProjectileManager`

### 원인

스크립트가 기대하는 객체가 존재하지 않거나, 실제 Explorer 이름과 코드의 참조 이름이 달랐습니다. 일부 객체는 생성 순서가 늦어 참조 시점에도 문제가 있었습니다.

### 확인 과정

- Explorer에 대상 객체가 실제로 존재하는지 확인
- 대소문자와 이름 일치 여부 확인
- 객체가 생성되는 서버 진입점과 참조 시점 비교
- `WaitForChild()` 호출 경로 확인

### 해결

누락된 Manager와 RemoteEvent를 추가하고, 생성 위치와 참조 이름을 일치시켰습니다.

### 배운 점

`WaitForChild()`는 누락된 객체를 해결해 주는 함수가 아니므로, 경고가 발생하면 먼저 Explorer 구조와 생성 순서를 확인해야 합니다.

## RemoteEvent queue exhausted

### 문제

서버가 RemoteEvent를 전송했지만 클라이언트에 대응하는 수신 연결이 없어 이벤트 큐 관련 경고가 발생했습니다.

### 확인된 사례

`QuestEvent`를 서버에서 전송했지만 클라이언트 수신 코드가 누락된 상태가 있었습니다.

### 해결

클라이언트에 필요한 `OnClientEvent` 수신 처리를 추가하고, 아직 사용하지 않는 이벤트는 불필요하게 전송하지 않도록 흐름을 점검했습니다.

### 배운 점

RemoteEvent를 추가할 때는 다음 흐름을 함께 확인해야 합니다.

```text
이벤트 생성
→ 서버 송신
→ 클라이언트 수신
→ UI 또는 상태 반영
```

## ClientController 문자열 구문 오류

### 문제

`ClientController`에서 다음과 같은 Lua 구문 오류가 발생했습니다.

```text
Malformed string
```

### 원인

긴 문자열 또는 줄바꿈을 포함한 UI 텍스트를 수정하는 과정에서 따옴표가 정상적으로 닫히지 않았습니다.

### 해결

오류가 발생한 줄 주변의 문자열 시작과 종료 문자를 확인하고, 줄바꿈이 필요한 부분을 명시적인 문자열 형식으로 수정했습니다.

### 배운 점

대형 단일 클라이언트 스크립트는 작은 UI 수정도 다른 구문의 오류로 이어질 수 있습니다. 장기적으로 HUD, 입력, 메뉴, 이펙트를 별도 모듈로 분리할 필요가 있습니다.

## 사운드 에셋 타입 불일치

### 문제

```text
Asset type does not match requested type
```

Sound 객체에 에셋 ID를 지정했지만 재생할 수 없었습니다.

### 원인

입력한 ID가 Sound 객체에서 사용할 수 있는 오디오 에셋이 아니거나 사용 권한이 맞지 않았습니다.

### 대응

- 에셋의 실제 타입 확인
- 오디오 공개 및 사용 권한 확인
- SoundId 형식 확인
- 문제가 있는 임시 사운드 제거 또는 교체

### 상태

오류 원인은 확인했으며, 최종 사운드 에셋 선정은 Visual Rebuild 단계에서 진행할 계획입니다.

## 직업과 무기가 즉시 갱신되지 않는 문제

### 문제

직업 변경 비용은 처리되지만 HUD의 직업 정보와 자동 공격 무기가 즉시 바뀌지 않는 경우가 있었습니다.

### 원인 추정

서버 데이터는 변경되었지만 클라이언트가 최신 직업과 장착 무기 상태를 다시 받지 못했거나, 기존 자동 공격 상태가 남아 있었을 가능성이 있습니다.

### 수정 방향

```text
클라이언트 직업 변경 요청
→ 서버 비용과 조건 검증
→ 서버 직업·장착 무기 상태 변경
→ 변경된 전체 상태를 클라이언트에 전달
→ HUD와 자동 공격 상태 재구성
```

### 상태

v2.7.2에서 수정했으며 실제 성공 여부는 회귀 테스트 대기 상태입니다.

## 이동속도와 사거리 아이템 적용 문제

### 문제

아이템 선택과 UI 표시는 동작하지만 다음 실제 스탯이 변하지 않는 경우가 있었습니다.

- 이동속도 선택 후 `Humanoid.WalkSpeed`
- 사거리 선택 후 공격 거리
- 사거리 표시 링의 크기와 실제 공격 거리

### 원인 추정

아이템 데이터 값, 서버 상태, 클라이언트 표시 값과 실제 전투 계산 값이 서로 다른 경로를 사용했을 가능성이 있습니다.

### 수정 방향

하나의 최종 스탯 계산 결과를 기준으로 이동, 공격 판정, HUD와 사거리 표시가 모두 같은 값을 사용하도록 정리합니다.

### 상태

수정 시도 후 재검증이 필요합니다.

## 진화 조건을 달성해도 진화하지 않는 문제

### 문제

필요한 무기와 패시브 조건을 갖춘 뒤에도 진화 카드가 나타나지 않거나 공격 방식이 변하지 않았습니다.

### 확인 항목

- 무기 레벨 조건
- 요구 패시브 보유 여부
- 진화 데이터의 ID 일치
- 레벨업 후보 생성 시 진화 우선순위
- 진화 후 기존 무기 교체와 공격 상태 갱신

### 상태

v2.7.2에서 조건 판정과 카드 생성 흐름을 수정했으며, 실제 플레이 재검증이 필요합니다.

---

## 문제 해결 원칙

1. 오류 메시지와 발생 줄을 먼저 확인합니다.
2. 코드 수정 전에 Explorer 구조와 객체 이름을 비교합니다.
3. 서버 데이터 변경과 클라이언트 UI 갱신을 별도로 확인합니다.
4. 구현 성공과 플레이 테스트 성공을 구분합니다.
5. 수정한 기능뿐 아니라 기존 기능도 회귀 테스트합니다.
6. 확인되지 않은 원인은 추정으로 기록하고, 테스트 후 확정합니다.
