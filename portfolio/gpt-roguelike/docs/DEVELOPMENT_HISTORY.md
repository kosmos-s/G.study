# 개발 과정

이 문서는 실제 개발 대화와 플레이 테스트를 바탕으로 GPT Roguelike가 어떻게 발전했는지 기록합니다. 과거 작업을 과장하거나 확인되지 않은 기능을 완료로 표시하지 않습니다.

## 1. 시작 계기

사람들이 바이브 코딩으로 웹사이트, 앱, 게임 등 다양한 결과물을 만드는 모습을 보면서 직접 무언가를 만들어 보고 싶다는 생각에서 시작했습니다.

그중 게임 제작을 선택했고, Unity처럼 새로운 엔진을 처음부터 배우기보다 예전에 YouTube 영상을 따라 사용해 본 경험이 있는 Roblox Studio를 선택했습니다. 익숙한 환경에서 AI를 활용해 실제로 플레이할 수 있는 게임 하나를 완성해 보는 것이 첫 목표였습니다.

## 2. v0.0.1 — 초기 게시

- Roblox 프로젝트 최초 게시
- 기본 공격과 적 추적
- 피해 처리 프로토타입
- 방 기반 로그라이크와 상점·무기 실험 시작

초기에는 기능을 빠르게 확인하는 데 집중했으며, 구조와 완성도보다 실제로 플레이 가능한 흐름을 먼저 만드는 방식으로 진행했습니다.

## 4. v2.5.1 — Visual Rebuild

- HUD와 메뉴 UI 개선 시도
- 공격·피격·사망 이펙트 확장
- 맵 조명과 환경 연출 실험
- 무기 레벨, 패시브 아이템, 진화 기반 추가
- 퀘스트·업적 기반 추가

기능은 늘어났지만 UI와 이펙트가 서로 다른 시기에 추가되면서 일관성이 부족해졌습니다. 이 과정에서 기능 구현과 실제 플레이 감각은 별개의 문제라는 점을 확인했습니다.

또한 서버에서 전송한 `QuestEvent`를 클라이언트가 수신하지 않아 RemoteEvent 큐 경고가 발생했고, `OnClientEvent` 연결을 함께 점검하는 방식으로 수정했습니다.

## 5. v2.6.0 — Mega RPG Economy

- 직업 5종
- 직업별 Q/E/R 스킬
- 직업 변경 비용과 직업별 무기
- RPG 스탯과 공격 사거리 표시
- 인게임 아이템 빌드와 진화
- 로비 상점
- F~S 무기 가챠
- 확률표, 천장, 중복 조각 구조

생존 전투에 직업, 장비, 경제 시스템이 연결되면서 프로젝트 규모가 크게 증가했습니다. 빠른 확장을 위해 많은 기능이 대형 Manager와 `ClientController`에 집중됐고, 이후 상태 동기화와 유지보수 문제가 나타났습니다.

## 6. v2.7.0 — Mega Systems Patch

- 원거리 몬스터 투사체
- 인벤토리와 장비 장착
- 판매
- 랭킹
- 플레이어 거래 프로토타입
- Robux 가챠 준비 UI

`ProjectileManager`와 일부 `RemoteEvents`가 누락되면서 서버 초기화와 맵 생성 흐름이 멈추는 문제가 발생했습니다. 필요한 객체의 존재 여부, 생성 순서, 이름 일치를 먼저 확인하는 방식으로 대응했습니다.

## 7. v2.7.1 — UX/Save/Job/Inventory Fix

- 최초 직업 선택 흐름 보완
- 직업 변경 비용 처리
- 레벨업 숫자키 1/2/3 입력
- 인벤토리 열기 수정

기능을 추가하는 것보다 이미 구현된 기능이 실제 플레이에서 즉시 반영되는지 검증하는 작업으로 중심이 이동했습니다.

## 8. v2.7.2 — Core State Fix

- 직업 변경 후 HUD와 무기 즉시 갱신 수정 시도
- 재접속 후 직업·장착 무기 복원 수정 시도
- 이동속도와 사거리 아이템 적용 수정 시도
- 진화 조건과 진화 카드 생성 흐름 수정 시도

코드 수정은 반영했지만 최근 회귀 테스트 기준으로 직업·무기 즉시 갱신, 이동속도, 실제 사거리, 진화, 재접속 저장은 재검증이 필요합니다.

다음 패치에서는 새로운 대형 기능보다 기반 안정화를 먼저 진행합니다.

- 게임 상태와 저장 데이터 동기화
- 직업·무기·스탯 변경 이벤트 흐름 통합
- Soul 보상 계산 점검

- Rojo 전환 준비
- `MapBuilder` 축소와 Studio 제작 맵 전환 준비

## 배운 점

1. 기능 수보다 서버 데이터와 클라이언트 화면의 상태 동기화가 중요했습니다.
2. 대형 단일 스크립트는 빠른 프로토타입에는 유리하지만 작은 수정도 회귀 오류로 이어질 수 있습니다.
3. `WaitForChild()` 경고가 발생하면 코드보다 먼저 Explorer 구조와 생성 순서를 확인해야 합니다.
4. 맵 전체를 런타임에 생성하면 초기 제작은 빠르지만 시각적 완성도를 높이기 어렵습니다.
5. 저장과 거래는 정상 흐름뿐 아니라 실패·이탈·중복 처리까지 설계해야 합니다.
6. AI가 제안한 코드는 실제 Roblox Studio 테스트를 거쳐야 프로젝트 상태로 인정할 수 있습니다.
7. 체크리스트를 이용하면 수정한 기능과 기존 기능의 회귀 여부를 함께 확인할 수 있습니다.
